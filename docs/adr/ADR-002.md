# ADR-002: Domain-Driven Design structure

## Status

Accepted

## Context

The codebase was organized by technical layer (controllers, services, repositories) making it hard to reason about agent-domain boundaries.

## Decision

Restructure into five bounded contexts: `agent-lifecycle`, `task-execution`, `memory-management`, `coordination`, `shared-kernel`. Each context owns its domain, application, infrastructure, and API layers.

## Consequences

### Positive

Clear ownership per domain. Enables independent scaling and testing. Aligns with how agents actually think about their work.

### Risks and mitigations

Higher initial setup cost. Teams must learn DDD vocabulary. Mitigated by this ADR and the DDD tracker.
